## 二、配置层详解

### 2.1 配置架构

文件：`config/settings.py`

采用 **Pydantic BaseSettings v2 + 单例模式 + .env 加载**。

```python
from functools import lru_cache
from pydantic_settings import BaseSettings, SettingsConfigDict

class LLMSettings(BaseSettings):
    # OpenAI 配置
    openai_api_key: str = ""
    openai_api_base: str = "https://api.openai.com/v1"
    llm_model: str = "gpt-4o-mini"
    embedding_model: str = "text-embedding-3-small"
    # Ollama 本地配置
    ollama_base_url: str = "http://localhost:11434"
    ollama_model: str = "qwen2.5:14b"
    use_local_model: bool = False
    model_config = SettingsConfigDict(env_prefix="OPENAI_", extra="ignore")

class Neo4jSettings(BaseSettings):
    uri: str = "bolt://localhost:7687"
    user: str = "neo4j"
    password: str = "kg_password_2024"
    database: str = "neo4j"
    model_config = SettingsConfigDict(env_prefix="NEO4J_", extra="ignore")

class ChromaDBSettings(BaseSettings):
    persist_directory: str = "./data/chroma_db"
    model_config = SettingsConfigDict(env_prefix="CHROMA_", extra="ignore")

class AppSettings(BaseSettings):
    host: str = "0.0.0.0"
    port: int = 8000
    debug: bool = True
    model_config = SettingsConfigDict(env_prefix="APP_", extra="ignore")

# 组合配置 + 单例
@lru_cache()
def get_settings():
    return AppSettings()
```

```dotenv

### 2.3 设计亮点

1. **env_prefix 自动绑定**：每个配置类设置不同的 `env_prefix`（如 `NEO4J_`），避免变量名冲突
2. **单例缓存**：`@lru_cache()` 装饰 `get_settings()`，全局只初始化一次
3. **双模型切换**：`USE_LOCAL_MODEL` 标志位控制在 OpenAI API 和本地 Ollama 之间无感切换
4. **类型安全**：Pydantic v2 自动类型校验 + 转换（如 "8000" → 8000）

### 3.2 图节点与生命周期状态机

文件：`models/graph_nodes.py`

**SkillNode（技能节点）**：
```python
class LifecycleStage(str, Enum):
    EMERGING = "emerging"      # 新兴（刚出现，<10%岗位提及）
    GROWING = "growing"        # 成长（热度上升，10-30%）
    MATURE = "mature"          # 成熟（主流技术，30-70%）
    DECLINING = "declining"    # 衰退（需求下降，<30%且降低）
    OBSOLETE = "obsolete"      # 过时（几乎无人提及）

class SkillNode(BaseModel):
    name: str
    domain: str                       # 所属领域（8大领域之一）
    category: str                     # 所属子分类
    description: str = ""
    lifecycle_stage: LifecycleStage = LifecycleStage.MATURE
    trend_score: float = 0.0          # -1.0 ~ 1.0
    confidence: float = 0.8           # 可信度

## 四、核心引擎层详解

### 4.1 图谱服务（GraphService）

文件：`core/graph_service.py`

**节点操作**：
- `create_skill_node(data)`：MERGE 技能节点
- `create_job_node(data)`：MERGE 岗位节点
- `create_domain_node(data)`：MERGE 领域节点
- `create_person_node(data)`：MERGE 人员画像节点

**关系操作**：
- `create_requires_relation(job_title, skill_name, weight)`：岗位-[REQUIRES]->技能
- `create_belongs_to_relation(skill_name, domain_name)`：技能-[BELONGS_TO]->领域
- `create_similar_relation(skill_a, skill_b, weight)`：技能-[SIMILAR_TO]->技能
- `create_has_skill_relation(person_id, skill_name, proficiency, credibility)`：人员-[HAS_SKILL]->技能

**图谱推理**：
- `get_job_required_skills(job_title)`：获取某岗位需要的所有技能（含权重）
- `get_person_skills(person_id)`：获取人员已掌握技能
- `find_similar_skills(skill_name, threshold)`：基于共同邻域+Jaccard相似度找相似技能
- `find_path_between_skills(start, end)`：Dijkstra 最短路径
- `get_subgraph_for_job(job_title, depth)`：获取某岗位相关子图

**管理方法**：
- `connect()` / `close()`：连接管理（含自动重连）
- `initialize_schema()`：创建索引/约束
- `seed_initial_domains()`：植入 8 大领域节点
- `get_graph_stats()`：节点/关系/按领域统计
- `execute_query(cypher, params)`：原始 Cypher 执行

**设计亮点**：
- `_get_session()` 方法实现惰性连接 + 自动重连，避免启动开销
- MERGE 语义：相同节点名重复写入不会创建重复，只会更新属性
- 所有节点包含 `last_updated` 时间戳，为时间序列分析提供基础

### 4.2 多源交叉验证引擎（MSVE）

文件：`core/cross_validator.py`

**核心机制 - 决策矩阵**：

| 一致性情况 | 置信度 | 处理方式 |
|-----------|--------|---------|
| 3 源一致 | 0.95 | 直接采纳，不进入审核 |
| 2 源一致 | 0.75 | 采纳，标记注意 |
| 单源 | 0.50 | 进入人工审核队列 |
| 冲突 | 0.25 | 触发 LLM 仲裁 |

**数据源基准置信度**：
```
SAMPLE: 1.00, MANUAL: 0.95, REPORT: 0.90
GITHUB: 0.75, BOSS_ZHIPIN: 0.80
LAGOU: 0.78, LIEPIN: 0.76, ZHAOPIN: 0.74
```

**主流程 validate_job(job_title)**：
1. 按岗位标题聚类 → `_cluster_key()`（title 小写 strip 作为 key）
2. **一致性分析** `_analyze_agreement()`：统计 skills/experience/salary/location 的众数率
3. **新鲜度因子** `_compute_freshness_factor()`：按采集日期衰减（>30天的衰减到0.5）
4. **源质量因子** `_compute_source_quality_factor()`：结合 source_base_confidence + 动态调整值
5. **加权数据融合** `_fuse_multi_source_data()`：
   - 薪资：加权平均（按源质量+新鲜度）
   - 技能：按出现频率+源质量加权
   - 经验年限：加权平均
   - 地点：加权众数
6. **冲突检测** `_detect_conflicts()`：同一字段跨源差异 > 30% 标记为冲突
7. **动态调整源置信度** `adjust_source_confidence()`：持续高冲突的数据源逐步降权

**输出结构**：

    "freshness_factor": 0.85,
    "final_confidence": 0.81,                 # 综合
    "fused_data": { "skills": [...], "salary_range": {...}, ... },
    "conflicts": [...],                       # 冲突列表

```

### 4.3 幻觉防控四层防御体系（HPF）

文件：`core/hallucination_guard.py`

#### L1 - 源头溯源（ProvenanceInfo）
每条三元组绑定溯源信息，确保所有图谱知识可追溯：
```python
class ProvenanceInfo:
    source_id: str            # 原始数据源ID
    source_type: str          # 来源类型（如"boss_zhipin"）
    record_id: str            # 原始记录ID
    raw_text: str             # 原始证据文本（截取前500字符）
    method: str               # 提取方法（rule/llm/hybrid）
    timestamp: str            # 时间戳
```

#### L2 - 约束引擎（ConstraintEngine）
四类约束检查，确保图谱结构合法：

1. **类型约束**：11 种合法关系类型白名单
   ```
   (Skill)-[similar_to]->(Skill)
   (Job)-[requires]->(Skill)
   (Job)-[prefers]->(Skill)
   (Skill)-[belongs_to]->(Domain)
   (Skill)-[part_of]->(Category)
   (Skill)-[evolves_to]->(Skill)
   (Skill)-[prerequisite_for]->(Skill)
   (Job)-[emerges_from]->(Job)
   (Person)-[has_skill]->(Skill)
   (Person)-[applied_for]->(Job)
   ```

2. **数值约束**：
   - difficulty ∈ [1, 10]
   - confidence ∈ [0.0, 1.0]
   - trend_score ∈ [-1.0, 1.0]
   - version 格式：`^\d+(\.\d+)?$`

3. **互斥规则**：
   - 同一岗位下，同一技能不能同时 required + optional
   - 岗位状态不能同时 active + archived
   - 技能不能同时属于两个不同主域

4. **入口方法** `validate_triple(head, relation, tail, properties)`：
   返回 `violations` 列表，非空则拒绝写入

#### L3 - 事实核查（FactChecker）
周期性一致性扫描，主动发现隐性问题：

- `run_consistency_scan()`：
  - **低置信度节点**：confidence < 0.30 → 标记 review
  - **孤儿节点**：无任何关系 → 提醒补充上下文
  - **异常簇**：同领域技能突然大量出现 → 触发人工复核
  - **过时节点**：>90 天未更新 → 推进生命周期状态机
  - **不一致**：同一岗位 required/optional 字段冲突

- `verify_single_claim(triple)`：对单个三元组做逆向验证（检查原始证据）

#### L4 - 人工审核队列（ReviewQueue）
- `add_for_review(item, reason)`：低置信度/冲突内容自动入队
- `approve(item_id)`：审核通过，自动提升 confidence
- `reject(item_id, reason)`：审核拒绝，自动删除
- `get_pending(limit)`：获取待审核列表
- `get_stats()`：审核统计（总数/通过率/平均处理时长）

#### 统一入口
```python
# 依次通过 L1 → L2 → L3 → L4
# 全部通过才允许写入图谱
def guard_triple(triple: GraphTriple, provenance: ProvenanceInfo) -> dict:
    result = {"passed": True, "warnings": [], "review_required": False}
    # L1: 溯源绑定（必做）
    # L2: 约束检查
    violations = constraint_engine.validate_triple(...)
    if violations:
        result["passed"] = False
        result["warnings"].extend(violations)
    # L3: 事实核查（低置信度）
    if triple.properties.get("confidence", 1.0) < 0.50:
        result["review_required"] = True
    # L4: 如需入队
    if result["review_required"]:
        review_queue.add_for_review(triple, "低置信度")
    return result
```

### 4.6 混合匹配引擎

文件：`core/match_engine.py`

**5 维加权打分公式**：
```
最终得分 = α × 技能匹配分      (默认权重 0.35)
         + β × 图谱路径分      (默认权重 0.20)
         + γ × 向量相似分      (默认权重 0.25)
         + δ × 趋势加分        (默认权重 0.10)
         + ε × 可信度修正      (默认权重 0.10)
```

**各维度实现**：

1. **技能匹配分** `_calc_skill_match()`：
   - 对 required_skills 逐项检查
   - alias 模糊匹配（如 "JavaScript" ↔ "JS"，"Python" ↔ "Python3"）
   - 隐含技能纳入计算（语义分析器的产出）
   - 公式：`matched_required / total_required × 0.8 + matched_optional / total_optional × 0.2`

2. **图谱路径分** `_calc_graph_distance()`：
   - 对每个 missing_required_skill，查找 nearest_resume_skill 的最短路径
   - 路径越短，说明 resume_skill 越容易迁移到 missing_skill
   - 公式：`avg_path_length → 归一化：1 / (1 + avg_path_length)`
   - 例子：缺失"Kafka"但掌握"Redis" → 路径 = 2 (Kafka→消息中间件→Redis) → 得 0.33

3. **向量相似分** `_calc_vector_similarity()`：
   - resume_embedding vs job_embedding 的余弦相似度
   - ChromaDB 原生支持，O(1) 查询

4. **趋势加分** `_calc_trend_bonus()`：
   - 使用岗位的 `trend_score`（-1.0 ~ 1.0）
   - 新兴岗位正向加权，鼓励候选人关注趋势
   - 公式：`0.5 + 0.5 × trend_score`

5. **可信度修正** `_calc_credibility_adjust()`：
   - 按技能可信度加权，而非同等对待
   - CERTIFIED(1.0) 的技能比 SELF_CLAIMED(0.2) 权重高 5 倍
   - 公式：`avg(matched_skill_credibility) / max_possible_credibility`

**输出对象**：
```python
@dataclass
class MatchResult:
    score: float                    # 0-100 的综合得分
    job_title: str
    breakdown: dict                 # 5 个维度的明细得分
    matched_skills: list[str]
    missing_critical: list[str]
    missing_optional: list[str]
    explanation: str                # 自然语言解释（LLM 生成）
```

**设计亮点**：通过 `generate_explanation()` 调用 LLM 将冷冰冰的数字转化为人类可读的解释，例如：
> "您在 Python 和 PyTorch 方面的项目经验直接命中该岗位核心需求（+35分），但缺少 Kubernetes 实操经验，建议通过3-6个月项目补齐（预计提升匹配率18%）。该岗位处于 Growing 阶段，市场需求上升趋势明显。"

### 4.8 向量服务

文件：`core/vector_service.py`

**3 个持久化集合**：
1. `skills`：技能描述 embedding → 用于相似技能检索
2. `jobs`：岗位需求 embedding → 用于岗位相似度匹配
3. `resumes`：简历画像 embedding → 用于简历相似度

**核心方法**：
- `add_vectors(ids, texts, metadatas)` → 批量入库
- `search(query_text, n_results, collection_name)` → Top-N 相似度检索
- `delete(ids)` → 删除
- `get_collection_stats(collection_name)` → 集合大小/维度统计

**设计**：懒加载客户端（首次调用时初始化），避免启动时阻塞。

### 4.9 数据管道

文件：`core/data_pipeline.py`

**ETL 流程**：
- `extract_fields(raw_json, source)` → 字段标准化映射
- `compute_completeness(record)` → 必填字段覆盖率（0-100）
- `compute_freshness(record_date)` → 数据新鲜度指数（0-100，按时间衰减）
- `deduplicate(records)` → 按 (title, company) 去重，保留置信度最高的
- `process_batch(raw_records)` → 批量 ETL，输出标准记录列表

## 五、多智能体层详解

### 5.1 Agent 基类架构

文件：`agents/base_agent.py`

每个 Agent 具备三大核心能力：**工具调用 + 记忆系统 + 自反思机制**。

#### 三层记忆系统（AgentMemory）

```
短期记忆（short_term: dict）
  → 当前任务上下文（如"当前解析的简历ID"）
  → 任务结束时通过 clear_short_term() 清空

长期记忆（long_term: list[dict]，保留最近100条）
  → 历史决策模式 + 成功/失败经验 + 教训
  → 通过 add_experience(task_type, decision, outcome, success, lesson) 写入
  → 通过 get_similar_experiences(task_type) 检索，避免重复踩坑

实体记忆（entity_memory: dict）
  → 跨会话实体跟踪（如 "某岗位第3次被更新"）
  → 记录每个实体的版本变更历史
  → 通过 track_entity(entity_id, entity_type, **attrs) 写入
  → 通过 get_entity_history(entity_id) 查阅历史
```

#### 自反思执行循环（_reflection_loop，最多 3 轮）

```
第1轮：
  perceive()   → 收集输入和上下文（含短期+长期记忆）
  hypothesize() → 生成候选方案（调用 LLM）
  validate()    → 检查约束/冲突（调用 ConstraintEngine）
  act()         → 执行方案（调用工具）
  reflect()     → 评估结果，若不佳则进入下一轮

第2轮：
  基于上一轮失败原因，重新 hypothesize()
  增加"避免X错误"的系统提示

第3轮：
  降级策略（如返回部分结果而非完全失败）

output() → 返回最终结果
```

#### 工具调用机制

每个 Agent 通过 `_setup_tools()` 注册 LangChain `BaseTool` 实例列表：
- `call_tool(tool_name, **inputs)` → 同步执行工具
- `think(prompt)` → 触发 LLM 推理（不操作数据）

### 5.2 编排中心 Agent

文件：`agents/orchestrator.py`

**职责**：接收用户请求 → 意图识别 → 任务分解 → 调度专业 Agent → 汇总结果 → 质量把关

**工作流**：
```python
def orchestrate(user_request, context):
    # Step 1: 意图识别（关键词匹配）
    intent = _analyze_intent(user_request)
    # 支持意图：collect_data / discover_jobs / build_graph
    #          / parse_resume / match_job / gap_analysis / quality_check

    # Step 2: 任务分解（按意图模板）
    subtasks = _decompose_task(intent, user_request)
    # 例：parse_resume → [解析文件 → 提取技能 → 语义分析 → 评分 → 入库]

    # Step 3: 分配Agent（按 subtask.type → agent_name 映射）
    for task in subtasks:
        agent = self._agent_registry[task.agent]
        results.append(agent.execute(task.input))

    # Step 4: 汇总 + 质量守护二次核查
    final = quality_guardian.verify_output(results)
    return final
```

**设计亮点**：编排中心本身不执行业务逻辑，只负责任务编排和质量把关，符合"单一职责"原则。

### 5.3 数据采集 Agent

文件：`agents/data_collector.py`

**核心能力**：从多源采集岗位数据，统一格式后交 MSVE 交叉验证。

**数据源优先级（SOURCES_PRIORITY）**：
1. Boss直聘 / 智联招聘 / 猎聘 / 拉勾网（招聘平台，最高）
2. GitHub Jobs（国际技术岗位）
3. 行业报告（人工整理，可信度高但更新慢）
4. 示例数据（SAMPLE，项目内置的演示数据）

**内置 26 条示例岗位数据**，覆盖 8 大领域：
- AI方向：AI算法工程师、大模型应用工程师、AI Agent开发工程师
- 大数据方向：大数据开发工程师、数据仓库工程师、实时计算工程师
- 云计算方向：云原生工程师、Kubernetes工程师、DevOps工程师
- 软件开发方向：全栈开发工程师、后端工程师、前端工程师、架构师

每条示例岗位含：技能列表、薪资范围、经验要求、城市分布、详细描述。

**采集方法**：
- `collect_all_sources()` → 并行采集所有源
- `collect_from_source(source_name)` → 单源采集
- `_get_sample_job_data()` → 返回内置 26 条示例（演示/无网络环境用）

### 5.4 新岗位发现 Agent

文件：`agents/job_discovery.py`

**核心能力**：从技能数据中识别"正在涌现但尚未有对应岗位"的新职业定义。

**发现流程 `discover_new_jobs()`**：
1. `_detect_emerging_skills()` → 识别新兴技能（trend_score > 0.7 且属于 Growing/Emerging 阶段）
   - 默认新兴技能列表：LangChain, LlamaIndex, LangGraph, RAG系统设计, Agent框架, Vector DB, LoRA微调, AI安全对齐, 多模态理解, 边缘AI
2. `_cluster_skills(emerging_skills)` → 按领域/技术栈聚类（如 "大模型应用栈"、"AI基础设施栈"）
3. `_detect_job_gaps(skill_clusters)` → 检查聚类是否已被现有岗位覆盖
   - 覆盖检查：聚类中 ≥ 60% 技能出现在某岗位的 required_skills 中则认为已覆盖
   - 未覆盖的聚类 → 产生候选新岗位
4. `_generate_job_definition(candidate)` → LLM 生成完整岗位定义（标题/职责/技能要求/经验/薪资预测）

**审核流程**：
- `approve_candidate(candidate_id)` → 审核通过，写入图谱为 JobNode
- 审核不通过 → 记录原因至长期记忆，后续类似候选降低优先级

### 5.5 图谱构建 Agent

文件：`agents/graph_builder.py`

**核心能力**：将原始数据转化为三元组写入 Neo4j，**每一次写入都经过 HPF 幻觉防控**。

**构建流程 `build_from_data(raw_records)`**：
1. `_process_job_record(record)` → 标准化为 JobNode
2. `_process_skills(job_node)` → 提取技能 + 绑定 provenance → 依次调用 `guard_triple()`
   - 通过 L1 溯源：绑定原始数据源信息
   - 通过 L2 约束：校验关系类型+数值+互斥
   - 通过 L3 事实：扫描异常模式
   - 通过 L4 审核：低置信度入队
3. `create_requires() / create_belongs_to()` → 写入图谱
4. `_vectorize_updates()` → 同步写入 ChromaDB（为向量匹配做准备）
5. `initialize_full_graph()` → 全量初始化（seed_initial_domains + 植入所有技能节点）

### 5.6 简历解析 Agent

文件：`agents/resume_parser.py`

**核心能力**：PDF/DOCX → 结构化画像（含 5 级可信度 + 隐含技能）。

**解析流程 `parse_resume(file_content)`**：
1. **文件解析** → FileParserTool（PyMuPDF for PDF / python-docx for DOCX）
2. **LLM 分段** → `structured_extraction(text, resume_schema)` 提取：
   - 基本信息（姓名/学历/工作年限）
   - 教育经历
   - 工作经历（公司/职位/时间/描述）
   - 项目经验（项目名/角色/技术栈/描述）
   - 技能清单
3. **显式技能提取** → 关键词匹配 + 分类体系对齐
4. **隐含技能推理** → `semantic_analyzer.infer_implicit_skills(project_descriptions)`
5. **可信度评估** → `credibility_scorer.assess_skill()` 逐项评估
6. **项目深度分析** → `evaluate_project_complexity()` 为每个项目打分
7. **向量入库** → `generate_embedding(resume_summary)` → ChromaDB resumes 集合
8. **输出画像** → `ResumeProfile` 对象

### 5.7 匹配 Agent

文件：`agents/matching_agent.py`

**核心方法**：
- `find_matches(resume_profile, job_titles=None)` → 获取全部岗位 → 批量 `calculate_match()` → 按得分排序返回 Top-N
- `simulate_what_if(resume_profile, additional_skills, target_job)` → What-If 模拟：
   - 创建"虚拟画像"（原技能 + 新增技能，可信度设为 INFERRED 0.65）
   - 重新计算匹配度
   - 通过 `_compare_match_results(before, after)` 输出对比（Δscore + 新增命中技能 + 剩余差距）

### 5.8 差距分析 Agent

文件：`agents/gap_analyzer.py`

**核心能力**：不仅识别差距，还要给**可执行的学习路径**和**ROI 分析**。

**流程 `full_gap_analysis(resume_profile, target_job)`**：
1. 识别 missing_critical / missing_optional
2. `_plan_learning_path(missing_skills)` → 对每个缺失技能：
   - **先序检测**：从图谱 prerequisite_for 关系找前置技能
   - **难度估算**：取 skill_node.difficulty
   - **资源推荐**：基于领域+难度，推荐课程/书籍/项目实践方案
   - **市场价值**：查该技能的 salary_weight + trend_score
3. `_analyze_learning_roi(path)` → ROI 计算：
   - 收益：学习后预计可申请岗位数量 × 平均薪资提升
   - 成本：预估学习时长（难度 × 40 小时）
   - 输出：ROI 比率 + 回收期

### 5.9 质量守护 Agent

文件：`agents/quality_guardian.py`

**核心能力**：全系统质量监控。

**`run_full_check()` 四个维度**：
1. **数据质量** `_check_data_quality()`：完整性、新鲜度、去重率
2. **图谱质量** `_check_graph_quality()`：孤儿节点率、低置信度率、关系密度
3. **合规性** `_check_compliance()`：幻觉防控触发率、审核队列积压

**`verify_output(output)` 最终把关**：
- 检查输出是否含低置信度内容
- 检查是否通过所有 HPF 层级
- 输出质量报告

## 十、完整数据流闭环 —— 以"用户上传简历 → 获取岗位推荐"为例

```
Step 1: 用户在前端上传 PDF 简历
  ↓ 前端：文件上传请求
Step 2: Backend POST /api/resume/upload
  ↓ 调用：FileParserTool → 解析为文本（PyMuPDF / python-docx）
  ↓ 调用：LLMService.structured_extraction → 结构化提取字段
  ↓ 调用：semantic_analyzer.infer_implicit_skills → 17 条规则 + LLM 推理
  ↓ 调用：credibility_scorer.assess_skill → 5 级可信度评估
  ↓ 调用：vector_service.add_vectors → 向量入库
  ↓ 返回：ResumeProfile（含 skills + implicit_skills + credibility + embedding_id）

Step 3: 用户选择岗位匹配
  ↓ Backend POST /api/match/match
  ↓ 调用：graph_service → 获取目标岗位的 required_skills + trend_score
  ↓ 调用：match_engine.calculate_match → 5 维加权打分
  ↓ （技能匹配分（0.35）+ 图谱路径分（0.20）+ 向量相似分（0.25）+ 趋势加分（0.10）+ 可信度修正（0.10）
  ↓ 返回：MatchResult 列表（Top-N 排序，附 explanation）

Step 4: 用户点击"差距分析"
  ↓ Backend POST /api/match/gap
  ↓ 调用：gap_analyzer._plan_learning_path → 识别 missing + 规划路径 + 先序检测 + 难度估算 + 资源推荐
  ↓ 调用：_analyze_learning_roi → ROI 计算（薪资提升 / 学习成本
  ↓ 返回：差距报告 + 学习路径 + ROI

Step 5: 用户选择"What-If 模拟
  ↓ Backend POST /api/match/whatif
  ↓ 调用：matching_agent.simulate_what_if → 创建虚拟画像 + 重新计算 + 对比输出
  ↓ 返回：before / after 对比（Δscore + 新增命中 + 剩余差距）

每一次写入图谱时，经过 HPF 幻觉防控：
  L1 源头溯源 → L2 约束校验 → L3 事实扫描 → L4 审核队列
每一批数据汇入时，经过 MSVE 多源交叉验证：
  聚类 → 一致性分析 → 加权融合 → 冲突检测 → 置信度调整
```

## 十一、核心创新点汇总

### 创新点 1：多智能体协同 (Multi-Agent Collaboration)

- **9 个专业 Agent + 1 个编排中心**：每个 Agent 具备记忆系统（短期/长期/实体三层）+ 自反思机制（最多 3 轮 perceive-hypothesize-validate-act-reflect 循环）
- **对比传统方式**：传统单一大模型推理对复杂任务容易出错；本方案通过专业分工（采集/发现/图谱/简历/匹配/差距/质量），每个Agent 专注于自己擅长的领域
- **文件**：`agents/base_agent.py`（基类）+ `agents/orchestrator.py`（编排）

### 创新点 2：时序动态知识图谱 (Time-Series Dynamic KG)

- **5 阶段生命周期状态机**：Emerging → Growing → Mature → Declining → Obsolete
- **动态更新机制**：每次采集后评估 trend_score 自动推进状态
- **趋势感知引擎**：在匹配引擎中作为 "趋势加分" 维度纳入打分公式
- **文件**：`models/graph_nodes.py`（模型定义）+ `core/graph_service.py`（图谱操作）

### 创新点 3：多源交叉验证引擎 (MSVE)

- **决策矩阵**：3 源一致 → 置信度 0.95，2 源一致 → 0.75，冲突 → 触发 LLM 仲裁
- **动态置信度调整**：持续表现不一致的数据源会被逐步降权
- **解决的问题**：单源数据噪声、招聘平台数据不一致、行业报告滞后
- **文件**：`core/cross_validator.py`

### 创新点 4：幻觉防控四层防御 (HPF)

- **L1 源头溯源**：每条三元组绑定原始数据源和原始证据文本
- **L2 约束引擎**：关系类型白名单 + 数值约束 + 互斥规则
- **L3 事实核查**：周期性一致性扫描（孤儿节点/低置信度/异常簇/过时节点）
- **L4 人工审核队列**：低置信度三元组进入人工审核队列
- **对比传统方案**：传统方案通常只做简单数据清洗，HPF 提供四层深度防控
- **文件**：`core/hallucination_guard.py`

### 创新点 5：语义分析器（超越关键词匹配）

- **17 条隐含技能推理规则**：从项目描述中推断未显式提及的技能（如"亿级流量"→"分库分表/CDN/读写分离"）
- **项目深度评估**：1-10 分多维度评分
- **红旗警示系统**：检测模糊语言/技术堆砌/时间线异常
- **文件**：`core/semantic_analyzer.py`

### 创新点 6：五维混合匹配引擎

- **5 维加权打分**：技能匹配 (0.35) + 图谱路径 (0.20) + 向量相似 (0.25) + 趋势加分 (0.10) + 可信度修正 (0.10)
- **可解释性输出**：通过 LLM 生成自然语言解释，而非仅输出冷冰冰的数字
- **对比传统方式**：传统仅做关键词命中或简单向量匹配，本方案融合结构推理 + 向量检索 + LLM 解释
- **文件**：`core/match_engine.py`
